Route a bug report to the repo that owns the defect. A bug in *shared
scaffolding* (the machinery project-init generated into this project) should be
filed **upstream** in project-init so it is fixed once for every project — not
re-reported per project. A bug in *this project's own code* is filed locally.

## Step 1 — Classify: project-level vs tooling/upstream

Decide who owns the faulty behavior. Use the heuristics; when a signal points
upstream, it is almost certainly upstream.

| Signal | Project-level → file **locally** | Tooling / scaffolding → file **upstream (project-init)** |
|---|---|---|
| Where the faulty file lives | the project's own `src/`, app code, business logic, its own tests | `.claude/hooks/`, `.claude/scripts/`, `.claude/skills/`, `.claude/rules/`, scaffolded `.github/workflows/*`, board automation, the DAG/command guard, governance policy, `config.yaml` machinery |
| Who wrote it | this project's team | generated by `project-init` scaffold/upgrade (you did not hand-write it here) |
| Would other project-init projects hit it? | No — specific to this repo | **Yes** — the same defect reproduces in any scaffolded project |
| Examples | "login endpoint returns 500", "pagination off-by-one", "our parser drops empty rows" | "the pre-commit gate blocks a valid commit", "board automation ignores `- Field:` bullet metadata", "the command guard false-positives on a safe command", "the issue-creation helper crashes with `--parent`", "a scaffolded rule file is wrong" |

If it is **project-level**, go to Step 2. If it is **tooling/upstream**, go to
Step 3. When genuinely unsure, ask the user one classifying question before
proceeding.

## Step 2 — Project-level bug: file locally (as normal)

File it in *this* repo the usual way:

- If this project has the GitHub lifecycle enabled, use the **`create_issue`
  skill** so the issue gets the project's own planning metadata and board fields.
- Otherwise create it however the project normally tracks work.

Confirm the title and body with the user, then stop — do not route upstream.

## Step 3 — Tooling bug: route upstream to project-init

### 3a. Resolve the upstream target (never hardcode)

Read the upstream repo from **`.claude/config.yaml`**, under the scaffold
record: `scaffold.variables.project_init_repo` (an `owner/repo` slug). For a
non-github.com fork, also read `scaffold.variables.project_init_host` (or
`project.project_init_host`) to build the right host.

```bash
grep -o '"project_init_repo": *"[^"]*"' .claude/config.yaml   # -> owner/repo
grep -o '"project_init_host": *"[^"]*"' .claude/config.yaml   # -> github.com | GHES host
```

If the record is absent (an older scaffold), fall back to the plugin
marketplace `repo` in `.claude/settings.json`, or ask the user for the
project-init repo. **Never hardcode a repo slug** — it differs per fork.

### 3b. Match the UPSTREAM repo's own conventions (not this template)

project-init's own issues do **not** use this project's scaffolded
`### Priority / Size / Area … Definition of Ready/Done` metadata block. Before
drafting, inspect a few of the upstream repo's recent issues and conform to what
you observe:

```bash
gh issue list -R <owner>/<repo> --limit 10 --json number,title,labels,body
```

(No `gh`/cross-repo read? `WebFetch` the repo's issues page instead.) project-init
conventionally uses:

- **Conventional-commit title prefixes**: `feat:` / `fix:` / `docs:` / `chore:` …
- **Plain numeric issue IDs** (no `PI-N` key in the title).
- **Label-based categorization**: `feature`, `enhancement`, `bug`,
  `documentation`, `help wanted`, `good first issue` — pick from labels that
  already exist upstream.
- A **lighter, prose body** (what happened · minimal repro · which shared
  component · why it affects every scaffolded project) — not the downstream
  Definition-of-Ready/Done checklist.

Detect and follow whatever the target repo actually does; the above is the
default when nothing contradicts it.

### 3c. Default filing method — a prefilled "new issue" link

The simple, zero-write-access route: build a `github.com` prefilled link the
user clicks to submit. This works even with no cross-repo permissions.

```
https://<host>/<owner>/<repo>/issues/new?title=<url-encoded-title>&body=<url-encoded-body>&labels=<comma-separated-labels>
```

URL-encode the title and body (spaces → `%20`, newlines → `%0A`, `#` → `%23`,
etc.). Present the link for the user to open and submit.

### 3d. Secondary method — direct create (only when scoped upstream)

Only if the current session is scoped to the upstream repo with write access
(e.g. you are running *in* project-init, or an MCP/`gh` token can write there),
create it directly instead of a link:

```bash
gh issue create -R <owner>/<repo> --title "<title>" --body "<body>" --label <label>
```

Create labels only if you have permission and the label genuinely doesn't
exist; otherwise drop the label rather than failing.

### 3e. Always confirm before filing

Before emitting the link or creating the issue, show the user the **target
repo**, **title**, and **body**, and get a go-ahead. Never file into another
repo silently.
