projects-orchestrator python · uv · apache-2.0 · read-only core

Cross-project orchestration layer for agentic development

One control surface for a whole fleet of agent-built projects.

A root orchestrator that sits above many repositories scaffolded with project-init. It reads each project's machine-readable self-description, runs that project's own gates, watches CI, cloud, drift and memory — and renders the fleet through four surfaces that can never disagree, all without ever writing a byte back into the projects it governs.

Read-only

It never writes the contract

The orchestrator only ever reads each child's descriptor, memory, and manifest. Mutations stay in review-gated CI — coupling is a named, versioned boundary, not shared internals.

Never-raise

A broken child degrades, never breaks the view

Every module touching a subprocess, file, or YAML returns degraded data instead of raising. Half-scaffolded, non-git, or timed-out projects become unknown/fail cells — the fleet table always renders.

Offline-first

Last-known results, cached

Check outcomes persist per (project, task) under $XDG_CACHE_HOME. Status answers "does it pass, and how fresh is that?" without re-running anything.

What it can do

The command surface, grouped by intent

Every data command accepts --json for external monitors, and exit codes are meaningful — checks exits non-zero when a gate fails, drift when a project diverged, doctor on non-conformance, upgrade-plan when a project is behind upstream.

The relationship

How it interacts with project-init

project-init scaffolds new repositories with a machine-readable self-description; the orchestrator reads that description across the whole fleet. The two tools are coupled only through a named, versioned descriptor contract — a boundary that lets each evolve independently.

project-init

upstream · the author
  • Scaffolds .claude/config.yaml into every new project
  • Records a scaffold.manifest of file hashes
  • Ships git hooks & a structured .claude/memory/
  • Cuts releases the fleet measures freshness against
──▶ reads contract writes back read-only coupling · additive-only versions

projects-orchestrator

downstream · the reader
  • Parses the descriptor into a ProjectDescriptor
  • Runs the project's declared gates, nothing guessed
  • Diffs the tree against the recorded manifest (drift)
  • Searches every project's memory as one corpus

Under the hood

Infrastructure & architecture

A never-raise engine feeds one shared view-model; every interface is a projection of it, so no two surfaces can disagree. Cross-repo I/O is quarantined in adapters, and per-project work fans out over a bounded pool.

Core engine

Reads contracts, never raises

descriptor.py parses the config; memory.py, drift.py, status.py and checks.py read the rest. Every subprocess flows through one timeout-bounded runner.py; unparseable input becomes defaults + warnings, not an exception.

Shared view-model

One truth, four surfaces

fleet.py joins descriptor + git status + cached checks + memory into a ProjectSnapshot. A single ordered COLUMNS set, the pure snapshot_row, and the presentation-free cell_status classifier decide what text every cell holds and whether it reads good / warn / bad — in exactly one place.

ProjectSnapshot → fleet_rows → CLI table --json Textual TUI HTML / serve
Adapters

Cross-repo I/O, quarantined

The only modules that reach the network sit behind a uniform seam: github / gitlab (CI + PRs), cloud (deploy state), and project_init (upstream releases + upgrade dispatch). Each still never raises and each caches its result so the fleet view stays offline.

Concurrency & state

Slowest project, not sum of projects

pool.py maps per-project work over a bounded thread pool (min(8, cpu)), preserving order and short-circuiting a single item to a plain loop. cache.py persists last-known results under $XDG_CACHE_HOME; a corrupt cache simply reads as empty.

reader

Contract reader, not author. No parallel metadata format is ever introduced.

degrade

The fleet view must render no matter what state a child is in.

declared

Gates are the child's own declared commands — an undeclared gate is skipped, never guessed.

offline

Only ci and cloud-status touch the network; every other command reads the cache.

one-edit

Adding a column is one entry in COLUMNS + a value in snapshot_row — every surface inherits it.

deterministic

The controller is LLM-free. /ask may only choose among existing intents; the dispatcher executes them.
